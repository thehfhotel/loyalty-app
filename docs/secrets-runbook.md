# Secrets & Deployment Runbook

Operational guide for managing secrets, deployment-time injection, rotation, and incident response. **Researchers reporting vulnerabilities should use [`SECURITY.md`](../SECURITY.md), not this file.**

## Principles

- All production secrets are stored in **GitHub Actions Secrets**, never in committed files.
- `.env` files are gitignored (except `.env.example`, which contains only placeholders).
- Production reads secrets from a `.env` file generated by the deploy workflow at deploy time.
- Use `${VAR}` substitution in `docker-compose.prod.yml`; never hard-code secrets there.

## Secret inventory

The following GitHub Actions secrets are expected to be configured under
**Settings → Secrets and variables → Actions → Repository secrets**:

| Secret Name             | Description                              |
| ----------------------- | ---------------------------------------- |
| `JWT_SECRET`            | JWT access token signing key             |
| `JWT_REFRESH_SECRET`    | JWT refresh token signing key            |
| `SESSION_SECRET`        | Session encryption key                   |
| `GOOGLE_CLIENT_ID`      | Google OAuth client ID                   |
| `GOOGLE_CLIENT_SECRET`  | Google OAuth client secret               |
| `LINE_CHANNEL_ID`       | LINE OAuth channel ID                    |
| `LINE_CHANNEL_SECRET`   | LINE OAuth channel secret                |
| `LOYALTY_USERNAME`      | First-run admin email                    |
| `LOYALTY_PASSWORD`      | First-run admin password (≥12 chars)     |

> Translation services are currently disabled. Azure translator secrets are not required unless that feature is re-enabled.

### Inspecting and updating secrets

```bash
# List configured secrets (values stay hidden)
gh secret list

# Set a secret from a literal string
gh secret set SECRET_NAME --body "secret-value"

# Set from a file
gh secret set SECRET_NAME < secret-file.txt

# Set interactively
gh secret set SECRET_NAME
# (paste value, then Ctrl+D)
```

## Deployment flow

```
GitHub Actions Secrets
        │
        ▼
.github/workflows/deploy.yml  (runs on push to main, approval for prod)
        │  injects secrets
        ▼
.env  (created on the runner / deploy host; gitignored, ephemeral)
        │  ${VAR} substitution
        ▼
docker-compose*.yml
        │
        ▼
Containers (backend, frontend, nginx, postgres, redis)
```

### `docker-compose.prod.yml` pattern

```yaml
services:
  backend:
    environment:
      JWT_SECRET: ${JWT_SECRET}
      GOOGLE_CLIENT_SECRET: ${GOOGLE_CLIENT_SECRET}
      # ...
```

No secrets are baked into images or compose files; everything flows from `.env`.

## Local development vs production

### Local development

```bash
cp .env.example .env
# Fill in dev-only credentials
docker compose up -d
```

### Automatic production deployment

```bash
git push origin main
# GitHub Actions:
#   1. Reads secrets from GitHub Secrets
#   2. Generates .env on the deploy host
#   3. Runs docker compose with the prod overlay
```

### Manual production deployment (recovery only)

```bash
cd <deploy-root>          # e.g. ${DEPLOY_ROOT}
cat > .env <<'EOF'
JWT_SECRET=...
JWT_REFRESH_SECRET=...
SESSION_SECRET=...
GOOGLE_CLIENT_ID=...
GOOGLE_CLIENT_SECRET=...
LINE_CHANNEL_ID=...
LINE_CHANNEL_SECRET=...
LOYALTY_USERNAME=...
LOYALTY_PASSWORD=...
EOF
chmod 600 .env

docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d
```

## Rotation procedure

1. **Generate a new value**

   ```bash
   # Random secrets
   openssl rand -hex 32

   # OAuth secrets: regenerate in the provider console
   ```

2. **Update the GitHub Secret**

   ```bash
   gh secret set JWT_SECRET --body "<new-value>"
   ```

3. **Trigger redeployment**

   ```bash
   git commit --allow-empty -m "chore: rotate JWT secret"
   git push origin main
   ```

### Recommended cadence

- JWT-family secrets: **every 90 days**
- OAuth credentials: when the provider requires, or **annually**
- API keys: **every 180 days**
- After any security incident: **immediately**

## Incident response: accidentally committed secret

If a secret is accidentally committed:

1. **Rotate the exposed secret immediately.** Treat any committed value as compromised, even if the commit was reverted.

   ```bash
   NEW_JWT=$(openssl rand -hex 32)
   gh secret set JWT_SECRET --body "$NEW_JWT"
   # Repeat for every exposed secret
   ```

2. **Redeploy** so the old value is no longer accepted:

   ```bash
   git commit --allow-empty -m "security: rotate exposed secrets"
   git push origin main
   ```

3. **Revoke at the provider** for OAuth and third-party API keys (Google Cloud Console, LINE Developers Console, etc.).

4. **Scrub history** if the secret was committed and you cannot tolerate the value remaining in `git log`. Use [`git filter-repo`](https://github.com/newren/git-filter-repo) (preferred) or BFG Repo-Cleaner. After scrubbing, force-push and notify all contributors to re-clone — coordinate carefully, as this rewrites history for everyone.

5. **Audit** GitHub access logs and application logs for unauthorized use of the exposed secret during its exposure window.

6. **Document** the incident: what was exposed, when, how it was detected, what was rotated, and what process change prevents recurrence.

## Secret-handling checklist

- [ ] No secrets in `docker-compose*.yml` (only `${VAR}` substitutions)
- [ ] `.env*` in `.gitignore` (except `.env.example`)
- [ ] All production secrets stored in GitHub Actions Secrets
- [ ] `LOYALTY_PASSWORD` is ≥12 chars, not a well-known default
- [ ] Rotation cadence followed (see above)
- [ ] Repository access restricted to authorized maintainers
- [ ] Deploy host has `.env` with `chmod 600` permissions

## CodeQL and log injection

CodeQL runs via GitHub's default setup (no custom workflow file) for SAST.

User-controlled values written to logs are sanitized with helper functions (newline / control-character / ANSI removal) to prevent log-injection (CWE-117). When CodeQL flags sanitized code as a false positive, dismiss with a clear comment naming the sanitizer in use:

```bash
gh api -X PATCH repos/OWNER/REPO/code-scanning/alerts/ALERT_NUMBER \
  -f state=dismissed \
  -f dismissed_reason="false positive" \
  -f dismissed_comment="Input is sanitized via the project's log-sanitizer helpers (newlines, CR/LF, control chars, ANSI escapes removed) per OWASP recommendations."
```

Valid `dismissed_reason` values: `"false positive"`, `"won't fix"`, `"used in tests"`.
